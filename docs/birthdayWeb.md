# 誕生日サプライズWebサイト 要件定義書

- 作成日: 2026-07-19
- バージョン: 1.1(管理画面・リアルタイム閲覧・通知を追加)
- 開発ツール: Claude Code

---

## 1. プロジェクト概要

### 1.1 目的

恋人の誕生日に、旅行そのものを「体験型のサプライズ」に変えるWebサイトを作る。QRコードを読み込むと手紙のようなオープニングが始まり、旅行中は各スポットでミッションが提示され、写真を撮って進めていく。旅の終わりには、集めた写真がそのまま「二人の思い出の本」になる。

### 1.2 利用者・利用シーン

| 項目 | 内容 |
|---|---|
| メインユーザー | 恋人(iPhone / Safari) |
| サブユーザー | 自分(プラン作成・写真の閲覧) |
| 利用シーン | 旅行当日、移動中・スポット到着時にスマホで操作 |
| アクセス方法 | 印刷したQRコード(手紙やカードに添付)→ URLを開く |
| 利用後 | 旅行後もURLを開けば思い出アルバムとして閲覧可能 |

### 1.3 コンセプト

- 「手紙 → 冒険 → 一冊の本」というストーリーで一貫させる
- デザインはディズニー/Appleのような、余白の効いたシンプルで上質な世界観
- 操作説明が不要なくらい直感的に(彼女は仕様を知らずに使う前提)

   │

   │
   ▼
[2] ホーム画面(進行状況マップ)◄──────┐
   │                                    │
   ▼                                    │

   ▼                                    │

## 3. 機能要件

### 3.1 合言葉ロック(初回アクセス時)

- QRコードのURLを開くと、最初に合言葉の入力を求める
- 合言葉は二人だけが分かるもの(例: 記念日、思い出の言葉)
- 一度認証したら localStorage に保存し、以降は再入力不要
- 誤入力時はエラー演出のみ(ロックアウトはしない)
- 目的はサプライズの保護であり、厳密なセキュリティではない

### 3.2 オープニング(手紙演出)

- 封筒が開く/本の表紙が開くようなアニメーションからスタート
- 「開く」ボタンのタップをトリガーにBGM再生を開始(iOS Safariの自動再生制限対策を兼ねる)
- 手紙の本文が1文字ずつ、または段落ごとにふわっと表示される
- 読み終えると「旅をはじめる」ボタンでホーム画面へ
- 2回目以降のアクセスではスキップ可能(ただしいつでも読み返せる)

### 3.3 ホーム画面(進行状況)

- 旅全体の進行状況を一目で確認できる
  - 案A: スタンプラリー風(達成スポットにスタンプ/写真サムネイルが埋まる)
  - 案B: 地図風の道のり表示(点と点が線でつながっていく)
- 現在挑戦中のミッションが強調表示され、タップで詳細へ
- 未来のミッションは「???」でロック表示(ネタバレ防止)
- 達成済みスポットはタップすると撮った写真を見返せる
- ヘッダーに「◯ / ◯ 達成」のカウンター表示

### 3.4 ミッション表示

- 1画面につき1ミッション。カード型でシンプルに表示
- 表示内容:
  - スポット名(または到着するまで伏せる演出も選択可)
  - ミッション本文(例: 「ここで二人のツーショットを撮ろう」)
  - ヒント(任意。折りたたみで表示)
  - メッセージ(任意。そのスポットに込めた一言)
- 「写真を撮る」ボタンでカメラ起動へ

### 3.5 写真撮影・アップロード

- `<input type="file" accept="image/*" capture="environment">` でiPhoneのカメラを直接起動(カメラロールからの選択も許可)
- アップロード前にクライアント側で画像を圧縮(目安: 長辺1600px / 500KB程度)→ 観光地の弱い電波でも送れるように
- アップロード中はプログレス表示、失敗時は自動リトライ+手動リトライボタン
- アップロード完了でミッション達成 → 達成演出(紙吹雪・スタンプが押されるアニメーション等)→ 次のミッションが解放
- 1ミッションにつき複数枚アップロード可(1枚目で達成扱い、追加は自由)
- 撮り直し(差し替え)可能

### 3.6 エンディング(思い出振り返り)

全ミッション達成後に解放される。構成は「本」のメタファーで統一する。

1. **表紙**: 「◯◯との旅の記録」のような表紙が現れ、タップで開く
2. **スライドショー**: 集めた写真が1枚ずつ、ゆっくりとしたズーム(Ken Burns風)+クロスフェードで流れる。各写真にスポット名・ミッション名・日時を小さく添える
3. **BGM**: スライドショーに合わせて再生(オープニングと同じ曲、またはエンディング専用曲)
4. **最後の手紙**: スライドショーの後、締めのメッセージが手紙形式で表示される
5. **アルバムへ**: 「アルバムを見る」ボタンで自由閲覧モードへ

### 3.7 アルバム(旅行後の閲覧)

- 全写真をグリッド表示、タップで拡大・スワイプで前後移動
- 本のようにページをめくって見るモードも用意(スワイプでページ送り)
- エンディングのスライドショーはいつでも再生し直せる
- 旅行後も同じURLでアクセス可能(思い出が残り続ける)

### 3.8 管理画面(自分専用)

彼女用の画面とは別URL(例: `/admin`)で管理画面を用意する。スマホ(自分のiPhone)からの操作を前提にモバイル対応で作る。

**認証**
- 管理者専用のパスワードでログイン(彼女用の合言葉とは別)
- 認証状態はlocalStorageに保持

**プラン編集**
- 旅行(trip)の作成・編集: タイトル、日付、合言葉、オープニングの手紙、エンディングの手紙
- スポット・ミッションの追加・編集・削除・並び替え(ドラッグ or 上下ボタン)
- 各スポットの項目: スポット名 / 名前を伏せるか / ミッション本文 / ヒント / 一言メッセージ
- **当日の編集に対応**: 保存すると彼女側の画面に即反映される(達成済みスポットの編集は警告を出す)
- 旅行を複数作成でき、旅行先ごとに使い回せる(可変式)

**進捗ダッシュボード(リアルタイム)**
- 現在の達成状況(◯/◯、いまどのミッションに挑戦中か)をリアルタイム表示
- アップロードされた写真が届いた順にフィードとして流れてくる(タップで拡大)
- Supabase Realtimeで自動更新(リロード不要)

**通知**
- 写真がアップロードされる(=ミッション達成)たびに、管理者にだけ通知が届く
- 実現方法: §8-7の決定により、メール通知を採用(下記Discord案は旧案)
- 通知内容: スポット名+達成時刻(+可能なら写真の署名付きURL)
- 彼女側には通知の存在を一切見せない

## 4. 非機能要件

| 分類 | 要件 |
|---|---|
| 対応端末 | iPhone(Safari)を最優先。画面幅 375〜430px を基準にデザイン |
| 表示速度 | 初回表示3秒以内(4G回線想定)。画像は遅延読み込み |
| 通信環境 | 弱電波を前提に、画像圧縮・リトライ・アップロード失敗時の状態保持を実装 |
| 進行状況の永続化 | サーバー(DB)に保存し、リロード・ブラウザ再起動でも消えない。localStorageは補助キャッシュ |
| セキュリティ | 合言葉ロック+推測されにくいURL(ランダムなパス)。写真は非公開バケットに保存し署名付きURLで表示 |
| 費用 | 無料枠内で運用(Vercel Hobby + Supabase Free) |
| プライバシー | 二人以外がアクセスできない構成。写真のEXIF位置情報はアップロード時に削除 |

## 5. 技術スタック(推奨案)

| レイヤー | 技術 | 理由 |
|---|---|---|
| フレームワーク | Next.js(React + TypeScript) | Vercelへのデプロイが最も簡単。Claude Codeの知見も豊富 |
| スタイリング | Tailwind CSS | モバイルファーストのUIを速く綺麗に作れる |
| アニメーション | Framer Motion | 手紙・ページめくり・達成演出などリッチな動きに強い |
| バックエンド | Supabase(Storage + Postgres) | 写真保存・プラン管理・進行状況が1サービスで完結。無料枠が広い |
| リアルタイム同期 | Supabase Realtime | 彼女の達成・写真アップを管理画面に即時反映 |
| 管理者通知 | Supabase Database Webhook + メール送信(Resend) | 写真INSERT時に管理者へメール通知(§8-7で決定) |
| 画像圧縮 | browser-image-compression | クライアント側圧縮の定番 |
| ホスティング | Vercel | 無料・自動デプロイ・HTTPS標準 |
| QRコード | qrcode ライブラリ or 外部サービスで生成し印刷 | 開発とは独立して用意可能 |
| BGM | 管理画面からアップロードした音源を使用(著作権上安全なものを選ぶ) | §8-4で決定 |

### データ構造(Supabase)

プランは管理画面から編集するため、JSONファイルではなくDBで管理する。

- `trips` テーブル: id / title / date / passphrase / opening_letter / ending_letter
- `spots` テーブル: id / trip_id / order / name / reveal_name / mission / hint / message
- `progress` テーブル: trip_id / spot_id / completed_at
- `photos` テーブル: trip_id / spot_id / storage_path / created_at
- Storage: `photos/{trip_id}/{spot_id}/{timestamp}.jpg`(非公開バケット)
- `photos` へのINSERTをDatabase Webhookで検知 → メール通知

## 6. デザイン方針

- **ベース**: 白・生成り系の背景に十分な余白。情報量を絞り、1画面1メッセージ
- **タイポグラフィ**: 見出しは明朝系(手紙らしさ)、本文はゴシック系で読みやすく。日本語Webフォントはサブセット化して軽量に
- **アクセント**: 1色だけテーマカラーを決める(薄い水色 / §8-6で決定)
- **モチーフ**: 手紙・封筒・スタンプ・本。装飾は最小限で、動き(アニメーション)で特別感を出す
- **アニメーション**: ゆっくり・柔らかく(ease-out、300〜600ms基調)。達成時だけ少し華やかに
- **ダークな画面**: スライドショーは黒背景で写真を主役に(映画のエンドロールのイメージ)

## 8. 決定事項(旧: 未確定事項)

1. **旅行の日程・行き先**: スポット数とプランの中身に直結(実装後に管理画面から投入)
2. **ホーム画面の表現**: スタンプラリー風
3. **スポット名を事前に見せるか**: 一部見せる(スポットごとに reveal_name で制御)
4. **BGMの選曲**: 管理者画面から好きな音楽をアップロード
5. **合言葉**: 記念日
6. **テーマカラー**: 薄い水色
7. **通知の送り先**: 管理者画面からメアド入力してメールに通知届くようにする
